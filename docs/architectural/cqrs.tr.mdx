## Overview

**Command Query Responsibility Segregation (CQRS)** is an architectural pattern that separates an application into two sides:

- **Command Side**: Handles write operations (mutations)
- **Query Side**: Handles read operations (retrievals)

This separation allows each side to be optimized independently for its specific purpose.

## Purpose

CQRS aims to:

- Optimize read and write operations independently
- Improve scalability by separating concerns
- Enable different data models for reads and writes
- Simplify complex domains
- Support different storage mechanisms for each side
- Enable independent scaling of read/write workloads

## Problem

Traditional CRUD models often suffer from:

- Same model used for reads and writes
- Write optimization conflicts with read optimization
- Complex queries require complex models
- Difficult to scale reads and writes independently
- Hard to optimize database queries
- Performance bottlenecks in either reads or writes

```
❌ Traditional CRUD Model
┌─────────────────────────┐
│   Single Unified Model  │
├─────────────────────────┤
│ • Add()                 │
│ • Update()              │
│ • Delete()              │
│ • GetAll()              │
│ • Search()              │
│ • Filter()              │
└─────────────────────────┘
  (Hard to optimize both)
```

## Solution

CQRS separates into two distinct models:

```
✅ CQRS Architecture
┌──────────────────────────────────────────────────────────┐
│                    Event Bus / Message Queue             │
└────────┬──────────────────────────────────┬──────────────┘
         │                                  │
┌────────▼──────────────────┐   ┌──────────▼─────────────┐
│   Write Model (Command)   │   │  Read Model (Query)    │
├───────────────────────────┤   ├──────────────────────┤
│ • Commands                │   │ • Query Objects      │
│ • Event Handlers          │   │ • Denormalized Data  │
│ • Validation              │   │ • Optimized Views    │
│ • Business Logic          │   │ • Cached Results     │
└───────────┬───────────────┘   └──────────┬──────────┘
            │                              │
     ┌──────▼──────┐              ┌───────▼────────┐
     │  Event Store│              │  Read Database │
     │  (Append-   │              │  (Optimized)   │
     │   Only)     │              └────────────────┘
     └─────────────┘
```

**Flow:**

1. Client sends command to write model
2. Write model processes and persists event
3. Event is published on event bus
4. Read model subscribers update denormalized data
5. Client queries read model for data

```typescript
// Events
interface Event {
  eventId: string;
  timestamp: Date;
  type: string;
}

interface UserCreatedEvent extends Event {
  type: "UserCreated";
  userId: number;
  name: string;
  email: string;
}

interface UserUpdatedEvent extends Event {
  type: "UserUpdated";
  userId: number;
  name?: string;
  email?: string;
}

// Commands
interface Command {
  commandId: string;
  timestamp: Date;
}

interface CreateUserCommand extends Command {
  name: string;
  email: string;
}

interface UpdateUserCommand extends Command {
  userId: number;
  name?: string;
  email?: string;
}

// Event Store (Write Model)
class EventStore {
  private events: Event[] = [];
  private subscribers: ((event: Event) => void)[] = [];

  appendEvent(event: Event): void {
    console.log(`[Event Store] Appended event: ${event.type}`);
    this.events.push(event);
    this.publish(event);
  }

  subscribe(callback: (event: Event) => void): void {
    this.subscribers.push(callback);
  }

  private publish(event: Event): void {
    this.subscribers.forEach((callback) => callback(event));
  }

  getEvents(): Event[] {
    return [...this.events];
  }
}

// Command Handler (Write Model)
class CommandHandler {
  private nextUserId = 1;
  private nextEventId = 1;

  constructor(private eventStore: EventStore) {}

  handleCreateUser(command: CreateUserCommand): void {
    console.log(`[Command Handler] Processing CreateUserCommand`);

    const event: UserCreatedEvent = {
      eventId: `event-${this.nextEventId++}`,
      timestamp: new Date(),
      type: "UserCreated",
      userId: this.nextUserId++,
      name: command.name,
      email: command.email,
    };

    this.eventStore.appendEvent(event);
  }

  handleUpdateUser(command: UpdateUserCommand): void {
    console.log(`[Command Handler] Processing UpdateUserCommand for user ${command.userId}`);

    const event: UserUpdatedEvent = {
      eventId: `event-${this.nextEventId++}`,
      timestamp: new Date(),
      type: "UserUpdated",
      userId: command.userId,
      name: command.name,
      email: command.email,
    };

    this.eventStore.appendEvent(event);
  }
}

// Read Model (Query Side)
interface UserReadModel {
  userId: number;
  name: string;
  email: string;
  createdAt: Date;
  updatedAt: Date;
}

class QueryModel {
  private users: Map<number, UserReadModel> = new Map();
  private usersByEmail: Map<string, number> = new Map();

  constructor(eventStore: EventStore) {
    eventStore.subscribe((event) => this.handleEvent(event));
  }

  private handleEvent(event: Event): void {
    if (event.type === "UserCreated") {
      const e = event as UserCreatedEvent;
      const user: UserReadModel = {
        userId: e.userId,
        name: e.name,
        email: e.email,
        createdAt: e.timestamp,
        updatedAt: e.timestamp,
      };
      this.users.set(e.userId, user);
      this.usersByEmail.set(e.email, e.userId);
      console.log(`[Query Model] Updated user ${e.userId}`);
    } else if (event.type === "UserUpdated") {
      const e = event as UserUpdatedEvent;
      const user = this.users.get(e.userId);
      if (user) {
        if (e.name) user.name = e.name;
        if (e.email) {
          this.usersByEmail.delete(user.email);
          user.email = e.email;
          this.usersByEmail.set(e.email, e.userId);
        }
        user.updatedAt = e.timestamp;
        console.log(`[Query Model] Updated user ${e.userId}`);
      }
    }
  }

  getUserById(userId: number): UserReadModel | undefined {
    return this.users.get(userId);
  }

  getUserByEmail(email: string): UserReadModel | undefined {
    const userId = this.usersByEmail.get(email);
    return userId ? this.users.get(userId) : undefined;
  }

  getAllUsers(): UserReadModel[] {
    return Array.from(this.users.values());
  }
}

// Application Facade
class UserApplication {
  private commandHandler: CommandHandler;
  private queryModel: QueryModel;

  constructor(eventStore: EventStore) {
    this.commandHandler = new CommandHandler(eventStore);
    this.queryModel = new QueryModel(eventStore);
  }

  createUser(name: string, email: string): void {
    const command: CreateUserCommand = {
      commandId: `cmd-${Date.now()}`,
      timestamp: new Date(),
      name,
      email,
    };
    this.commandHandler.handleCreateUser(command);
  }

  updateUser(userId: number, name?: string, email?: string): void {
    const command: UpdateUserCommand = {
      commandId: `cmd-${Date.now()}`,
      timestamp: new Date(),
      userId,
      name,
      email,
    };
    this.commandHandler.handleUpdateUser(command);
  }

  getUserById(userId: number): UserReadModel | undefined {
    return this.queryModel.getUserById(userId);
  }

  getAllUsers(): UserReadModel[] {
    return this.queryModel.getAllUsers();
  }
}

// Usage
const eventStore = new EventStore();
const app = new UserApplication(eventStore);

app.createUser("John Doe", "john@example.com");
app.createUser("Jane Smith", "jane@example.com");

console.log("\n=== Query Results ===");
const users = app.getAllUsers();
console.log("All users:", users);

app.updateUser(1, "John Updated", "john.updated@example.com");

console.log("\n=== Updated Query Results ===");
console.log("User 1:", app.getUserById(1));
```

```python
from abc import ABC, abstractmethod
from typing import List, Optional, Callable, Dict
from dataclasses import dataclass
from datetime import datetime

# Events
@dataclass
class Event:
    event_id: str
    timestamp: datetime
    event_type: str

@dataclass
class UserCreatedEvent(Event):
    user_id: int
    name: str
    email: str

@dataclass
class UserUpdatedEvent(Event):
    user_id: int
    name: Optional[str] = None
    email: Optional[str] = None

# Commands
@dataclass
class Command:
    command_id: str
    timestamp: datetime

@dataclass
class CreateUserCommand(Command):
    name: str
    email: str

@dataclass
class UpdateUserCommand(Command):
    user_id: int
    name: Optional[str] = None
    email: Optional[str] = None

# Event Store (Write Model)
class EventStore:
    def __init__(self):
        self.events: List[Event] = []
        self.subscribers: List[Callable] = []

    def append_event(self, event: Event) -> None:
        print(f"[Event Store] Appended event: {event.event_type}")
        self.events.append(event)
        self.publish(event)

    def subscribe(self, callback: Callable[[Event], None]) -> None:
        self.subscribers.append(callback)

    def publish(self, event: Event) -> None:
        for callback in self.subscribers:
            callback(event)

    def get_events(self) -> List[Event]:
        return self.events.copy()

# Command Handler (Write Model)
class CommandHandler:
    def __init__(self, event_store: EventStore):
        self.event_store = event_store
        self.next_user_id = 1
        self.next_event_id = 1

    def handle_create_user(self, command: CreateUserCommand) -> None:
        print(f"[Command Handler] Processing CreateUserCommand")

        event = UserCreatedEvent(
            event_id=f"event-{self.next_event_id}",
            timestamp=datetime.now(),
            event_type="UserCreated",
            user_id=self.next_user_id,
            name=command.name,
            email=command.email
        )
        self.next_user_id += 1
        self.next_event_id += 1
        self.event_store.append_event(event)

    def handle_update_user(self, command: UpdateUserCommand) -> None:
        print(f"[Command Handler] Processing UpdateUserCommand for user {command.user_id}")

        event = UserUpdatedEvent(
            event_id=f"event-{self.next_event_id}",
            timestamp=datetime.now(),
            event_type="UserUpdated",
            user_id=command.user_id,
            name=command.name,
            email=command.email
        )
        self.next_event_id += 1
        self.event_store.append_event(event)

# Read Model (Query Side)
@dataclass
class UserReadModel:
    user_id: int
    name: str
    email: str
    created_at: datetime
    updated_at: datetime

class QueryModel:
    def __init__(self, event_store: EventStore):
        self.users: Dict[int, UserReadModel] = {}
        self.users_by_email: Dict[str, int] = {}
        event_store.subscribe(self.handle_event)

    def handle_event(self, event: Event) -> None:
        if event.event_type == "UserCreated":
            e = event
            user = UserReadModel(
                user_id=e.user_id,
                name=e.name,
                email=e.email,
                created_at=e.timestamp,
                updated_at=e.timestamp
            )
            self.users[e.user_id] = user
            self.users_by_email[e.email] = e.user_id
            print(f"[Query Model] Updated user {e.user_id}")
        elif event.event_type == "UserUpdated":
            e = event
            if e.user_id in self.users:
                user = self.users[e.user_id]
                if e.name:
                    user.name = e.name
                if e.email:
                    self.users_by_email.pop(user.email, None)
                    user.email = e.email
                    self.users_by_email[e.email] = e.user_id
                user.updated_at = e.timestamp
                print(f"[Query Model] Updated user {e.user_id}")

    def get_user_by_id(self, user_id: int) -> Optional[UserReadModel]:
        return self.users.get(user_id)

    def get_all_users(self) -> List[UserReadModel]:
        return list(self.users.values())

# Application Facade
class UserApplication:
    def __init__(self, event_store: EventStore):
        self.command_handler = CommandHandler(event_store)
        self.query_model = QueryModel(event_store)

    def create_user(self, name: str, email: str) -> None:
        command = CreateUserCommand(
            command_id=f"cmd-{int(datetime.now().timestamp())}",
            timestamp=datetime.now(),
            name=name,
            email=email
        )
        self.command_handler.handle_create_user(command)

    def update_user(self, user_id: int, name: Optional[str] = None, email: Optional[str] = None) -> None:
        command = UpdateUserCommand(
            command_id=f"cmd-{int(datetime.now().timestamp())}",
            timestamp=datetime.now(),
            user_id=user_id,
            name=name,
            email=email
        )
        self.command_handler.handle_update_user(command)

    def get_user_by_id(self, user_id: int) -> Optional[UserReadModel]:
        return self.query_model.get_user_by_id(user_id)

    def get_all_users(self) -> List[UserReadModel]:
        return self.query_model.get_all_users()

# Usage
if __name__ == "__main__":
    event_store = EventStore()
    app = UserApplication(event_store)

    app.create_user("John Doe", "john@example.com")
    app.create_user("Jane Smith", "jane@example.com")

    print("\n=== Query Results ===")
    users = app.get_all_users()
    print("All users:", users)

    app.update_user(1, "John Updated", "john.updated@example.com")

    print("\n=== Updated Query Results ===")
    print("User 1:", app.get_user_by_id(1))
```

## Real-World Examples

### E-Commerce Platform

- **Commands**: ProcessOrder, UpdateInventory
- **Queries**: GetProductDetails, ListAvailableProducts
- Maintains separate read cache for fast queries

### Social Media

- **Commands**: PostTweet, LikePost, CreateComment
- **Queries**: GetTimeline, GetFollowers, SearchHashtags
- Read model heavily denormalized for performance

### Banking System

- **Commands**: Deposit, Withdraw, Transfer
- **Queries**: GetBalance, GetTransactionHistory
- Strict separation for compliance and audit trails

## Advantages ✅

- **Independent Optimization**: Read and write paths optimized separately
- **Scalability**: Read and write workloads can scale independently
- **Performance**: Query model can be heavily optimized (caching, denormalization)
- **Simpler Queries**: Read model tailored for specific queries
- **Auditability**: All changes captured as events
- **Event Sourcing Ready**: Natural fit with event sourcing
- **Complex Domains**: Better handling of complex business logic
- **Independent Teams**: Read and write teams can work independently

## Disadvantages ❌

- **Complexity**: Increased architectural complexity
- **Learning Curve**: Developers need to understand CQRS
- **Consistency**: Eventual consistency between read and write models
- **Synchronization**: Keeping models in sync can be challenging
- **Testing**: More complex testing scenarios
- **Debugging**: Harder to trace issues across models
- **Storage**: Requires storage for both models
- **Overhead**: Extra infrastructure and processes

## When to Use ✅

- **High-Volume Systems**: With distinct read/write patterns
- **Complex Domains**: With intricate business rules
- **Scalability Critical**: When independent scaling needed
- **Reporting-Heavy**: With significant reporting requirements
- **Event-Driven**: Already using event sourcing
- **Microservices**: Different services handling reads/writes
- **High-Performance Reads**: When query optimization critical
- **Audit Requirements**: Needing complete audit trails

## When NOT to Use ❌

- **Simple CRUD Applications**: Over-engineering simple apps
- **Balanced Read/Write**: Similar load on both
- **Small Teams**: Increased complexity not justified
- **Real-Time Consistency**: Requiring strict consistency
- **Prototypes**: Quick throwaway projects
- **Simple Domains**: Straightforward business logic
- **Rapid Development**: Where speed is critical
- **Microservices Early**: Complex for first services

## Related Patterns

- **Event Sourcing**: Often used together with CQRS
- **Mediator Pattern**: For separating commands and queries
- **Observer Pattern**: For event subscription
- **Command Pattern**: For command handling
- **Memento Pattern**: For event snapshots
- **Saga Pattern**: For distributed transactions in CQRS
