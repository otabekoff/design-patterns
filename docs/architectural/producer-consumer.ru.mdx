## Overview

The **Producer-Consumer** pattern decouples the production of data from its consumption using a buffer or queue. Producers add items to the buffer, and consumers remove and process them independently.

Key concepts:

- **Producer**: Creates data/work items
- **Consumer**: Processes data/work items
- **Buffer/Queue**: Intermediate storage
- **Decoupling**: Independent rates of production/consumption

## Purpose

Producer-Consumer aims to:

- Decouple production from consumption
- Handle different production/consumption rates
- Improve system throughput
- Enable parallel processing
- Reduce resource blocking
- Support asynchronous workflows

## Problem

Without decoupling:

- Producers blocked waiting for consumers
- Consumers blocked waiting for producers
- Inefficient resource utilization
- Hard to balance workload
- Performance bottlenecks

```
❌ Direct Producer-Consumer
Producer → blocked → Consumer
```

## Solution

Buffer decouples producer and consumer:

```
✅ Producer-Consumer with Buffer
Producer → [Buffer/Queue] → Consumer
         (Independent rates)
```

```typescript
// Queue implementation
class Queue<T> {
  private items: T[] = [];

  enqueue(item: T): void {
    this.items.push(item);
  }

  dequeue(): T | undefined {
    return this.items.shift();
  }

  isEmpty(): boolean {
    return this.items.length === 0;
  }

  size(): number {
    return this.items.length;
  }
}

// Work item
interface WorkItem {
  id: number;
  data: string;
}

// Producer
class DataProducer {
  constructor(private queue: Queue<WorkItem>) {}

  produce(count: number): void {
    for (let i = 0; i < count; i++) {
      const item: WorkItem = {
        id: i,
        data: `Item ${i}`,
      };
      this.queue.enqueue(item);
      console.log(`✅ Produced: ${item.data}`);
    }
  }
}

// Consumer
class DataConsumer {
  constructor(private queue: Queue<WorkItem>) {}

  consume(maxItems: number = 10): void {
    let count = 0;
    while (!this.queue.isEmpty() && count < maxItems) {
      const item = this.queue.dequeue();
      if (item) {
        this.processItem(item);
        count++;
      }
    }
  }

  private processItem(item: WorkItem): void {
    console.log(`👤 Consumed: ${item.data}`);
  }
}

// Usage
const queue = new Queue<WorkItem>();
const producer = new DataProducer(queue);
const consumer = new DataConsumer(queue);

console.log("=== Producing items ===");
producer.produce(5);

console.log("\n=== Consuming items ===");
consumer.consume();

console.log(`\nQueue size: ${queue.size()}`);
```

```python
from typing import TypeVar, Generic, List, Optional
from dataclasses import dataclass
import time

T = TypeVar('T')

# Queue implementation
class Queue(Generic[T]):
    def __init__(self):
        self.items: List[T] = []

    def enqueue(self, item: T) -> None:
        self.items.append(item)

    def dequeue(self) -> Optional[T]:
        return self.items.pop(0) if self.items else None

    def is_empty(self) -> bool:
        return len(self.items) == 0

    def size(self) -> int:
        return len(self.items)

# Work item
@dataclass
class WorkItem:
    id: int
    data: str

# Producer
class DataProducer:
    def __init__(self, queue: Queue):
        self.queue = queue

    def produce(self, count: int) -> None:
        for i in range(count):
            item = WorkItem(id=i, data=f"Item {i}")
            self.queue.enqueue(item)
            print(f"✅ Produced: {item.data}")

# Consumer
class DataConsumer:
    def __init__(self, queue: Queue):
        self.queue = queue

    def consume(self, max_items: int = 10) -> None:
        count = 0
        while not self.queue.is_empty() and count < max_items:
            item = self.queue.dequeue()
            if item:
                self.process_item(item)
                count += 1

    def process_item(self, item: WorkItem) -> None:
        print(f"👤 Consumed: {item.data}")

# Usage
if __name__ == "__main__":
    queue: Queue[WorkItem] = Queue()
    producer = DataProducer(queue)
    consumer = DataConsumer(queue)

    print("=== Producing items ===")
    producer.produce(5)

    print("\n=== Consuming items ===")
    consumer.consume()

    print(f"\nQueue size: {queue.size()}")
```

## Real-World Examples

- **Thread Pools**: Threads produce/consume tasks
- **Message Queues**: RabbitMQ, Kafka
- **Data Pipelines**: ETL processes
- **Web Servers**: Request/response handling
- **Stream Processing**: Spark, Flink

## Advantages ✅

- **Decoupling**: Independent production/consumption
- **Throughput**: Better system performance
- **Scalability**: Easy to add producers/consumers
- **Buffer**: Handles rate mismatches
- **Flexibility**: Different pacing
- **Responsive**: Reduces blocking
- **Load Balancing**: Distribute work
- **Resilience**: Handle spikes

## Disadvantages ❌

- **Complexity**: More code needed
- **Memory**: Buffer consumes memory
- **Latency**: Queue adds delay
- **Ordering**: May not preserve order
- **Debugging**: Harder to trace issues
- **Overflow**: Buffer can overflow
- **Underflow**: Queue can underflow
- **Synchronization**: Complex threading

## When to Use ✅

- **Parallel Processing**: Multiple threads
- **Rate Mismatches**: Different speeds
- **Asynchronous Work**: Decoupled tasks
- **Buffering**: Smoothing workload
- **Thread Pools**: Work queues
- **Message Systems**: Message queues
- **Real-Time Systems**: Handling bursts
- **Data Pipelines**: ETL processes

## When NOT to Use ❌

- **Simple Sequential**: One-at-a-time
- **Real-Time Critical**: Latency matters
- **Limited Memory**: Buffer too expensive
- **Ordering Critical**: Must preserve order
- **Simple Tasks**: Over-engineering
- **Low Throughput**: No benefit
- **Synchronous API**: Must return immediately
- **Single Threaded**: No concurrency

## Related Patterns

- **Observer Pattern**: Event notification
- **Mediator Pattern**: Central communication
- **Command Pattern**: Queued commands
- **Thread Pool Pattern**: Producer-consumer variant
